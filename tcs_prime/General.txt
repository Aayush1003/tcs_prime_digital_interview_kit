CONSOLIDATED TOPIC SUMMARIES
========================================

DSA (Data Structures & Algorithms)
- Core Topics: Arrays, LinkedList, Stack, Queue, Trees (BST/AVL), Graphs, Heaps, Tries, HashMaps, Sorting, Searching, Dynamic Programming.
- Big-O Notation: Describes asymptotic upper bound; common examples: O(1), O(log n), O(n), O(n log n), O(n^2). Use Big‑O for comparing scalability.
- Binary Search: Works on sorted arrays by halving search space. Iterative complexity O(log n) time, O(1) extra space. Recursive uses O(log n) stack. Example (Python):
```
def binary_search(a, target):
	 lo, hi = 0, len(a)-1
	 while lo <= hi:
		  mid = (lo+hi)//2
		  if a[mid]==target: return mid
		  if a[mid] < target: lo = mid+1
		  else: hi = mid-1
	 return -1
```
- Sorting (Quick vs Merge): Quick Sort average O(n log n), worst O(n^2), in-place but unstable; Merge Sort stable, always O(n log n), needs O(n) extra space. Choose Merge for stability/guarantee, Quick for in-place with good average case.
- Stack / Queue: Stack LIFO (use for DFS, call stack). Queue FIFO (BFS, job scheduling). Implement with arrays or linked lists — push/pop amortized O(1).
- HashMap / Hash Table: Average O(1) lookup; collisions via chaining or open addressing; resizing keeps load factor bounded.
- Dynamic Programming: Identify overlapping subproblems + optimal substructure. Use memoization (top-down) or tabulation (bottom-up). Typical problems: knapsack, LIS, Fibonacci, DP on trees/strings.
- Interview approach (DSA): Clarify requirements → give brute-force → optimize (time/space) → write clean code → test edge cases → state complexity and trade-offs.

Database (SQL / NoSQL)
- SQL vs NoSQL: SQL = relational, ACID, structured schemas (Postgres/MySQL). NoSQL = key-value/document/column-family/graph (Mongo/Cassandra), better for unstructured data and horizontal scaling.
- Normalization: 1NF (atomic values), 2NF (no partial dependency on PK), 3NF (no transitive dependency), BCNF (every determinant is a candidate key). Use normalization to reduce redundancy; denormalize for read performance when needed.
- Joins & Clauses: INNER JOIN (intersection), LEFT JOIN (keep left), RIGHT JOIN (keep right), FULL OUTER (both). WHERE filters rows before aggregation; HAVING filters groups after aggregation.
- Indexes: B-tree indexes for range queries, hash indexes for equality; indexes speed reads, slow writes and use space. Use composite and covering indexes to optimize queries. Use EXPLAIN to inspect plans.
- Transactions & ACID: Atomicity, Consistency, Isolation, Durability. Mention isolation levels (Read Uncommitted → Read Committed → Repeatable Read → Serializable) and trade-offs (performance vs anomalies).
- Triggers / Stored Procedures: Triggers run on table events; procedures encapsulate logic on DB side — use judiciously for consistency, avoid complex business logic in DB if it hurts portability.
- Sample SQL (2nd highest salary):
```
-- using subquery
SELECT MAX(salary) FROM employees WHERE salary < (SELECT MAX(salary) FROM employees);

-- using window function
SELECT salary FROM (
  SELECT salary, DENSE_RANK() OVER (ORDER BY salary DESC) r FROM employees
) t WHERE r = 2;
```

Web Technologies (REST, HTTP, JSON, MVC)
- REST API: Resource-based URIs, stateless, use correct HTTP verbs (GET/POST/PUT/PATCH/DELETE), return meaningful status codes (200/201/204/400/401/403/404/500). Document via OpenAPI/Swagger.
- HTTP vs HTTPS: HTTPS = HTTP over TLS (encryption + server auth). Always use HTTPS in production.
- HTTP Methods: GET (safe, idempotent), POST (create, not idempotent), PUT (replace, idempotent), PATCH (partial update), DELETE (idempotent).
- JSON vs XML: JSON preferred for REST: lighter, native to JS; XML used where schemas/complex documents are needed.
- Auth & Security: Sessions/cookies vs token-based (JWT). Protect against CSRF, XSS, SQL injection; use input validation and parameterized queries.
- CORS & Preflight: Browser-enforced cross-origin policy; configure server Access-Control-Allow-* headers; preflight OPTIONS check for non-simple requests.
- MVC / Architecture: Model = data, View = UI, Controller = request logic. For large systems discuss layering, caching, rate-limiting, and monitoring.

Operating Systems (OS)
- OS role: Manages hardware, processes, memory, file systems, I/O, and provides abstractions.
- Process vs Thread: Process = independent address space; thread = lightweight execution unit sharing memory. Threads cheaper to create, faster context switch, but need synchronization.
- Deadlock: Occurs when four conditions hold: mutual exclusion, hold-and-wait, no preemption, circular wait. Prevent via resource ordering, avoid hold-and-wait, or use timeouts and deadlock detection with rollback.
- Scheduling: FCFS (simple), SJF (minimize average wait), Round Robin (fair, uses time quantum), Priority (may starve low-priority). Know preemptive vs non-preemptive differences.
- Concurrency primitives: Mutex (binary lock), Semaphore (counting), Condition variables, Monitors. Use them to synchronize shared state and avoid race conditions.

Networking
- OSI Model (7 layers): Physical (bits), Data Link (frames, MAC), Network (routing, IP), Transport (end-to-end, TCP/UDP), Session (sessions), Presentation (format/encoding), Application (HTTP/DNS). Be able to map TCP/IP stack to OSI.
- TCP vs UDP: TCP = reliable, ordered, connection-oriented; UDP = unreliable, low-latency, connectionless (useful for streaming, DNS).
- IP / Subnet / Gateway: IP = host address; subnet mask defines network bits; gateway routes packets outside local network. Know CIDR notation (e.g., /24).
- DNS: Translates domain names to IPs via recursive/iterative queries; caches at resolver and client. Understand A/AAAA/CNAME/MX records.
- Packets & Ports: Packet = unit of data at network layer; ports identify application endpoints (e.g., 80/443). Explain NAT basics if asked.

HR / Behavioral
- Tell me about yourself: 30–60s pitch: education → 1–2 key technical strengths → one major project → what you want next. Keep it role-focused.
- Strengths & Weaknesses: Strength = concrete example + impact. Weakness = real shortcoming + steps taken to improve. Avoid clichés.
- Behavioral answers (STAR): Situation, Task, Action, Result — use metrics when possible. Example: “Reduced test runtime by 40% by refactoring X; result: Y.”
- Why TCS / Relocation / Salary: Show company research, align skills to role, express flexibility on location and reasonable salary expectations; give a range based on market research.
- Questions to ask interviewer: Team structure; success metrics for role; onboarding/training; current technical challenges.
